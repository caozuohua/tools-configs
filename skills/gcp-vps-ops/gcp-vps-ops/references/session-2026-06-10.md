# Session Reference: GCP SSH Troubleshooting (2026-06-10)

## Context
- **Source instance:** gcp-vps2 (current machine, 7.7GB RAM)
- **Target instance:** instance-20260413-080555 in us-central1-c (internal IP 10.128.0.3, 1GB RAM)
- **Service account:** 40129528744-compute@developer.gserviceaccount.com (default GCE service account)
- **Goal:** Establish SSH access from source to target for Hermes deployment evaluation

## Error Timeline & Resolution

### 1. Direct SSH Failure
```bash
ssh -i ~/.ssh/google_compute_engine caozuohua99@10.128.0.3
# Permission denied (publickey)
```
**Reason:** OS Login manages keys via metadata, not authorized_keys.

### 2. gcloud compute ssh - Missing compute.instances.get
```bash
gcloud compute ssh instance-20260413-080555 --zone=us-central1-c
# ERROR: Required 'compute.instances.get' permission
```
**Fix:** Granted `roles/compute.instanceAdmin.v1` to service account.

### 3. gcloud compute ssh - Missing iam.serviceAccountUser
```bash
gcloud compute ssh instance-20260413-080555 --zone=us-central1-c
# ERROR: The user does not have access to service account...
# Ask a project owner to grant you the iam.serviceAccountUser role
```
**Fix:** Granted `roles/iam.serviceAccountUser` to service account.

### 4. Success
```bash
gcloud compute ssh instance-20260413-080555 --zone=us-central1-c --command="echo test"
# gcloud SSH 连接成功
```

## Additional Techniques Discovered (Session Continuation)

### Remote Python Execution
- Running Python via `gcloud compute ssh --command` requires careful quote handling
- Single quotes for outer wrapper, double quotes for inner Python strings
- For complex scripts, upload via `gcloud compute scp` then execute

### Systemd User Service + Linger (on target)
- `hermes-gateway.service` runs as systemd user service
- Must enable linger to survive logout: `sudo loginctl enable-linger $USER`
- Verified with: `loginctl show-user $USER --property=Linger`

### Git Permission Debugging (on source)
- `/opt/luck-agent` owned by `luck-agent` user, current user in `luck-agent` group
- `.git/refs/heads/main` owned by root → no write access without sudo
- No sudo, no password for luck-agent user → cannot pull from current user
- Requires root intervention: `sudo chown -R caozuohua99:luck-agent /opt/luck-agent/.git/refs`
