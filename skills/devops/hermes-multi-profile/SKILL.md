# Hermes Multi-Profile Diagnosis & Config Mirror

## When to use

The user's primary `~/.hermes/` home has been customized, but a Hermes
service (systemd unit or `hermes gateway install --run-as-user`) is
configured with `Environment=HERMES_HOME=/path/to/secondary-home`
(e.g. `~/.hermes-lite/`) and the service therefore reads from the
secondary home. Symptoms:

- Service is `active/running` and connected to its platform
  (Lark / Discord / Slack / etc.), but LLM calls fail with
  `Primary provider auth failed: No inference provider configured`
- `/model`, `/status`, `/help`, or other slash commands return
  default / canned / "原始状态" responses
- The user has been editing the primary home (`~/.hermes/config.yaml`,
  `~/.hermes/memories/MEMORY.md`, `~/.hermes/skills/qpc/`, etc.) and
  wonders why the running service doesn't pick up the changes
- A recent edit to a systemd unit file added or removed
  `Environment=HERMES_HOME=...`, and the symptoms started around
  the next service restart (NOT the edit time — see Pitfalls)

## Quick diagnosis (5 SSH checks)

```bash
# 1. Find active HERMES_HOME
gcloud compute ssh INSTANCE --zone=ZONE --command='
  systemctl show <unit> -p Environment | grep HERMES_HOME'

# 2. Compare config sizes + missing blocks
gcloud compute ssh INSTANCE --command='python3 -c "
import yaml
m=yaml.safe_load(open(\"/home/USER/.hermes/config.yaml\"))
l=yaml.safe_load(open(\"/home/USER/.hermes-lite/config.yaml\"))
print(f\"primary: {len(m)} keys, secondary: {len(l)} keys\")
print(f\"in primary, NOT in secondary: {sorted(set(m)-set(l))[:30]}\")
"'

# 3. Check memories + skills
gcloud compute ssh INSTANCE --command='
  ls /home/USER/.hermes-lite/memories/ /home/USER/.hermes-lite/skills/'

# 4. When did the service last start? (HERMES_HOME change is dormant until then)
gcloud compute ssh INSTANCE --command='
  systemctl show <unit> -p ActiveEnterTimestamp'

# 5. When did the unit file last change? (compare to ActiveEnterTimestamp)
gcloud compute ssh INSTANCE --command='
  stat /etc/systemd/system/<unit>.service | grep Modify'
```

If the unit file's mtime is **earlier** than `ActiveEnterTimestamp`,
the file was edited AFTER the last restart and the change WILL take
effect on the next restart. If the unit file's mtime is **later** than
`ActiveEnterTimestamp`, the change is already live.

## Fix (4-step mirror)

1. **Backup** everything you'll touch in the secondary home
2. **Push + run** `scripts/config-mirror.py` to overlay config, mirror
   memories, mirror user-local skills
3. **Restart** the service (`sudo systemctl restart <unit>`)
4. **Verify** Lark WS reconnects + memory peak stays under cap

See `references/hermes-home-gotcha.md` for the full procedure with
edge cases, and `scripts/config-mirror.py` for the reusable template.

## Pitfalls

- **HERMES_HOME change is dormant until next service restart** — the
  unit-file edit + service NOT restarted = change is invisible. The
  cause can be hours/days before the symptom. Always compare unit
  file mtime to `ActiveEnterTimestamp`.

- **Platform-specific config blocks** (discord, telegram, slack,
  whatsapp, mattermost, matrix) MUST be skipped when overlaying.
  Secondary profile may use a different platform (e.g. lite = feishu,
  primary = discord). The user would notice double-platform or
  wrong-platform behavior if you don't skip.

- **`disabled_toolsets` lives in the `agent:` block** — keep
  secondary's value (likely tighter than primary's, for memory
  reasons on small VPS).

- **USER.md may have been auto-generated by the running agent** — it
  writes a system profile (HERMES_HOME / model / disabled toolsets)
  on first start. If `<HERMES_HOME>/memories/USER.md` mtime is < 24h
  old, BACK IT UP and ASK the user before overwriting. Sometimes the
  user wants to append (USER profile + system profile), sometimes
  overwrite (USER profile only — the system profile is auto-derived).

- **Two bak files end up similar** — use a UTC timestamp suffix
  `bak-pre-sync-YYYYMMDD-HHMMSS` to disambiguate from
  `bak-pre-merge-YYYYMMDD-HHMMSS` (the first config merge for the
  model block).

- **bash heredoc + Python f-string with nested quotes breaks** — use
  `write_file` to push the script first, then `gcloud ssh ... --command='python3 /tmp/SCRIPT.py'`
  to run it. Don't inline Python in `--command='...'`.

- **Don't push the user's primary `~/.hermes/` config to a remote
  via local Hermes's `write_file` / `patch`** — cross-host isolation
  protocol (see gcp-vps-ops skill memory). Do the config mirror ON
  the remote via SSH, not from local.
